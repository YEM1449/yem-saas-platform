# Security Baseline

This document summarizes the security mechanisms visible in the current implementation.

## Authentication Modes

The platform uses two bearer-token modes:

| Mode | Principal | Role set | Main routes |
| --- | --- | --- | --- |
| CRM JWT | `app_user` ID | `ROLE_SUPER_ADMIN`, `ROLE_ADMIN`, `ROLE_MANAGER`, `ROLE_AGENT` | `/auth/*`, `/api/*` |
| Portal JWT | buyer `contact` ID | `ROLE_PORTAL` | `/api/portal/*` |

## CRM JWT Claims

Current CRM tokens are generated by `JwtProvider` and validated by `JwtAuthenticationFilter`.

| Claim | Meaning |
| --- | --- |
| `sub` | user UUID |
| `roles` | Spring Security roles array |
| `tv` | token version for revocation |
| `sid` | active societe UUID, omitted for platform-only `SUPER_ADMIN` tokens |
| `partial` | present on short-lived societe-selection tokens |
| `imp` | optional impersonating super-admin UUID |

Important:

- older docs referring to `tid` are outdated for the current implementation
- partial tokens are only valid for `POST /auth/switch-societe`

## Portal Authentication

Portal auth is a two-stage flow:

1. `POST /api/portal/auth/request-link` stores a SHA-256 hash of a random token and sends a link.
2. `GET /api/portal/auth/verify` validates and consumes that one-time token, then issues a 2-hour `ROLE_PORTAL` JWT.

Portal details confirmed in code:

- magic-link storage TTL: 48 hours
- portal JWT TTL: 2 hours
- portal JWT contains `sub=contactId`, `sid=societeId`, `roles=["ROLE_PORTAL"]`
- portal JWTs do not carry `tv`

## Request Security Flow

`SecurityConfig` and `JwtAuthenticationFilter` enforce the main request boundary.

### Public routes

- `/auth/login`
- `/auth/switch-societe`
- `/auth/invitation/**`
- `/actuator/health`
- `/actuator/info`
- `/swagger-ui/**`
- `/swagger-ui.html`
- `/v3/api-docs/**`
- `/api/portal/auth/request-link`
- `/api/portal/auth/verify`

### Role-gated route groups

- `/api/admin/**` -> `SUPER_ADMIN`
- `/api/societes/**` -> `SUPER_ADMIN` legacy alias
- `/api/portal/**` -> `ROLE_PORTAL`
- `/api/**` -> CRM roles only

### Filter behavior

For CRM requests:

- validates signature and expiry
- rejects partial tokens outside `/auth/switch-societe`
- validates `enabled` and `tokenVersion` through `UserSecurityCacheService`
- sets `SocieteContext`

For portal requests:

- validates signature and expiry
- skips `UserSecurityCacheService`
- sets `SocieteContext` using the buyer contact as principal

## Authorization Model

### Platform tier

- `SUPER_ADMIN` manages societes, can impersonate members, and can assign the `ADMIN` role inside a societe.

### Company tier

- `ADMIN`
- `MANAGER`
- `AGENT`

Important rule:

- only `SUPER_ADMIN` can assign `ADMIN`
- a company `ADMIN` can assign only `MANAGER` or `AGENT`

This is enforced by `SocieteRoleValidator` and supported by integration tests.

## Token Revocation

CRM tokens are revocable before expiry.

Mechanism:

- `app_user.token_version` is embedded in JWT claim `tv`
- role changes, disable actions, password resets, and some membership changes increment token version
- `JwtAuthenticationFilter` compares the token claim to cached server state

Caching:

- Caffeine by default
- Redis when `REDIS_ENABLED=true`

Operational impact:

- revocation is near-immediate, bounded by cache refresh timing

## Rate Limiting and Lockout

### Login rate limiting

Two login buckets are enforced before password verification:

- per-IP bucket
- per-identity bucket

Current configuration keys:

- `RATE_LIMIT_LOGIN_IP_MAX`
- `RATE_LIMIT_LOGIN_KEY_MAX`
- `RATE_LIMIT_LOGIN_WINDOW_SECONDS`

### Portal magic-link rate limiting

The portal link request flow uses `RateLimiterService.checkPortalLink(...)`.

Current configuration keys:

- `RATE_LIMIT_PORTAL_CAPACITY`
- `RATE_LIMIT_PORTAL_REFILL_PERIOD`

### Account lockout

User lockout uses:

- `failed_login_attempts`
- `locked_until`

Config keys:

- `LOCKOUT_MAX_ATTEMPTS`
- `LOCKOUT_DURATION_MINUTES`

## Data Isolation

Application-layer isolation is the primary control:

- services resolve the active societe from `SocieteContext`
- clients do not supply societe scope directly for CRM resources
- portal services further constrain reads to the current buyer contact

Database defense in depth:

- `RlsContextAspect` sets PostgreSQL session variable `app.current_societe_id`
- RLS is currently enabled only on `contact` and `property`

## Transport and Browser Security

Implemented controls include:

- stateless sessions
- CSRF disabled for bearer-token APIs
- configurable CORS allowlist
- CSP with self-only defaults and inline styles allowed
- `X-Frame-Options: DENY`
- HSTS only when embedded TLS is enabled in the application

## GDPR and Data Protection Controls

Evidence in code includes:

- contact consent fields and privacy notice endpoint
- contact export and anonymization
- user export and anonymization
- anonymization blocked when signed contracts still require identity retention
- internal societe notes excluded from public-safe DTOs and user GDPR exports

## Confirmed Security-Relevant Gaps

The following gaps were observed in code, not assumed:

- Societe suspension fields exist, but no request-time enforcement of `SOCIETE_SUSPENDED` was found in login or domain services.
- Resource quota fields exist for biens, contacts, and projets, but enforcement was only confirmed for user quotas.
- `SocieteContext` contains a `role` slot, but no current request path populates it.
- `SecurityConfig` still reserves `POST /tenants`, although no active tenant-bootstrap controller was found.
- Springdoc is configured at `/api-docs`, while the public security matcher still references `/v3/api-docs/**`.
